### M2-T03 [feature] 任务管理模块与 API

- **模块**：任务管理
- **状态**：待开发

**描述**

实现任务的查询和状态机流转，对应接口设计第三章（不含执行相关接口）。

**验收标准**

- `GET /api/projects/{projectId}/tasks` 支持 status / type 过滤，按 updatedAt 降序
- `GET /api/tasks/{taskId}` 返回完整任务详情
- `POST /api/tasks/{taskId}/cancel` 仅允许 pending / in_progress / pending_review 状态流转
- `POST /api/tasks/{taskId}/reset` 仅允许 failed 状态流转，清空 branch / worktreePath 字段
- 非法状态流转返回 40901

### M2-T04 [feature] Git 服务模块

- **模块**：执行引擎
- **状态**：待开发

**描述**

封装 Git 操作，供执行引擎调用。所有 Git 命令通过 Python subprocess 调用本地 git CLI。

**验收标准**

- `create_branch(repo_path, branch_name, base="dev")` 从 dev 切出新分支
- `create_worktree(repo_path, worktree_path, branch_name)` 创建 worktree
- `remove_worktree(repo_path, worktree_path)` 删除 worktree
- `commit_and_push(worktree_path, message)` 在 worktree 内提交并推送
- 任意步骤失败时抛出包含 git stderr 的异常，调用方统一处理

---

### M2-T05 [feature] Prompt 构建器

- **模块**：执行引擎
- **状态**：待开发

**描述**

根据任务信息和经验总结，构造发送给 Claude Code 的完整 Prompt 文本。

**验收标准**

- 输出包含：任务类型/标题/模块/描述、验收标准列表、分支和 worktree 路径说明
- 若 `经验总结.md` 非空，将其全文附加到 Prompt 中
- Prompt 末尾包含明确的结果格式要求：`SUCCESS` 或 `FAILURE: <原因>`
- Prompt 为纯文本字符串，无多余格式

---

### M2-T06 [feature] 进程管理器

- **模块**：执行引擎
- **状态**：待开发

**描述**

负责 Claude Code 子进程的启动、监控和终止，维护进程池。

**验收标准**

- `start(task_id, worktree_path, prompt)` 以 worktree 为工作目录启动 `claude --print <prompt>` 进程
- 进程池以 `task_id` 为键存储运行中的进程句柄
- `stop(task_id)` 终止指定进程并从进程池移除
- 进程启动失败时抛出 50003 对应异常
- 进程意外退出时触发回调，通知执行引擎处理状态

---

### M2-T07 [feature] 输出解析器与执行引擎整合

- **模块**：执行引擎
- **状态**：待开发

**描述**

异步监听 Claude Code 进程 stdout，解析结果标记，驱动任务状态机。整合 Git 服务、进程管理、Prompt 构建为完整的执行引擎。

**验收标准**

- 逐行读取 stdout，识别最后一行的 `SUCCESS` 或 `FAILURE: <原因>` 标记
- `SUCCESS`：更新任务状态为 `pending_review`
- `FAILURE`：更新任务状态为 `failed`，调用 FileWriter 追加经验总结
- 进程意外退出（无结果标记）：状态置为 `failed`，写入异常信息到经验总结
- `POST /api/tasks/{taskId}/execute` 触发完整流程：校验状态 → 创建分支 → 创建 worktree → 构建 Prompt → 启动进程 → 状态置为 in_progress，接口立即返回
- 取消任务时同步终止进程并调用 `remove_worktree`

---

### M2-T08 [feature] 日志服务模块

- **模块**：日志服务
- **状态**：待开发

**描述**

将 Claude Code 进程输出持久化为 JSONL 文件，并通过 SSE 实时推送到前端。

**验收标准**

- 每行 stdout 写入 `data/logs/{task-id}.jsonl`，格式 `{"ts": ..., "stream": "stdout", "line": ...}`
- `GET /api/tasks/{taskId}/logs` 返回历史日志数组，按时间升序
- `GET /api/tasks/{taskId}/logs/stream` 建立 SSE 连接，进程运行中实时推送，推送格式符合接口设计第四章定义
- 进程结束后发送 `event: done`，任务已结束时订阅可直接返回历史日志后关闭

### M2-T09 [feature] 文件写入模块

- **模块**：文件写入
- **状态**：待开发

**描述**

向被管理项目的文档文件追加经验总结，以及生成 Bug 报告文件。

**验收标准**

- `append_experience(project_path, task_title, failure_reason)` 向 `docs/开发文档/经验总结.md` 追加符合 PRD 3.6 格式的内容
- `create_bug_report(project_path, task, failed_criteria)` 在 `docs/bug/` 下生成 Bug 报告，文件名格式 `<YYYY-MM-DD>-<模块>-<功能>-<简短描述>.md`，内容符合 PRD F-12 模板
- `read_experience(project_path)` 读取 `经验总结.md` 全文，文件不存在时返回空字符串
- `docs/bug/` 目录不存在时自动创建

### M2-T10 [feature] Review 接口与提交流程

- **模块**：任务管理
- **状态**：待开发

**描述**

实现 `POST /api/tasks/{taskId}/review` 接口，处理测试通过和测试失败两条路径。

**验收标准**

- result=pass：调用 `commit_and_push` → `remove_worktree` → 任务状态置为 completed，响应含 branch 信息
- result=fail：调用 `create_bug_report` → 创建新 bug 类型任务 → 原任务状态置为 failed，响应含 bugReportPath 和 newBugTaskId
- result=fail 且 failedCriteria 为空时返回 40001
- 前置状态校验：非 pending_review 状态返回 40901

- **模块**：文件写入
- **状态**：待开发

**描述**

扫描项目 `docs/bug/` 目录，解析 Bug 报告文件头，返回摘要列表。

**验收标准**

- `GET /api/projects/{projectId}/bugs` 扫描 `docs/bug/*.md`
- 解析文件名得到 createdAt、module、feature、description
- 解析文件头元数据（时间、模块、功能、任务 ID）
- 按 createdAt 降序返回，目录不存在时返回空数组

### M4-T06 [feature] useLogStream 组合函数（SSE 日志流）

- **模块**：前端组件
- **状态**：待开发

**描述**

封装 SSE 日志流订阅逻辑，供任务详情页 LogViewer 使用。

**验收标准**

- 组件挂载时建立 `EventSource` 连接

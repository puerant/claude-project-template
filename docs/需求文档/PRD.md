> 状态：草稿

## 一、产品概述

### 1.1 背景

在使用 Claude Code 进行项目开发时，AI 会生成结构化的任务清单（Markdown 格式）。当前这些任务清单缺乏系统化管理，无法做到：任务状态追踪、自动触发 AI 执行、执行结果归档、失败经验沉淀。

**Claude Code Manager** 旨在解决上述问题，成为 Claude Code 驱动开发流程的管控中枢。

### 1.2 产品定位

面向使用 Claude Code 进行 AI 辅助开发的团队或个人，提供**任务可视化管理 + AI 自动执行 + 质量闭环**的全流程工具。

### 1.3 核心价值

- 将 Markdown 任务清单结构化，实现任务状态的精确追踪
- 自动触发 Claude Code 执行任务，减少人工干预
- 建立失败经验知识库，避免重复踩坑
- 测试驱动的质量闭环，保障交付质量

## 三、核心功能需求

### 3.1 项目管理

#### F-01 添加项目

- 用户可添加一个本地项目文件夹路径
- 系统读取该路径下 `docs/开发文档/任务清单.md` 文件
- 每个项目独立管理，互不干扰

#### F-02 项目列表

- 展示所有已添加的项目
- 显示项目名称、路径、任务统计（总数/各状态数量）

---

### 3.2 任务清单解析

#### F-03 解析任务清单.md

系统读取指定项目的 `docs/开发文档/任务清单.md`，将其解析为结构化的 `任务清单.json`。

解析规则：

- 以 Markdown 标题（`##` 或 `###`）为任务边界进行分割
- 识别任务类型标记：`[feature]` 或 `[bug]`
- 识别任务状态标记（见 3.3）
- 提取任务描述、验收标准、关联模块等字段

任务 JSON 数据结构：

```json
{
  "tasks": [
    {
      "id": "uuid",
      "type": "feature | bug",
      "title": "任务标题",
      "description": "任务完整描述（Markdown 原文）",
      "module": "所属模块",
      "status": "pending",
      "acceptanceCriteria": ["验收条件1", "验收条件2"],
      "branch": null,
      "worktreePath": null,
      "createdAt": "ISO8601时间",
      "updatedAt": "ISO8601时间",
      "executionLog": null,
      "bugReportPath": null
    }
  ]
}
```

#### F-04 同步任务状态

- 支持手动触发重新解析，合并更新（不覆盖已有状态）
- 新增任务追加，已有任务保留执行状态不重置

---

### 3.3 任务状态管理

#### F-05 任务状态定义

| 状态       | 标识             | 说明                         |
| ---------- | ---------------- | ---------------------------- |
| 待开发     | `pending`        | 任务已创建，尚未开始         |
| 开发中     | `in_progress`    | Claude Code 正在执行         |
| 待 review  | `pending_review` | 执行完成，等待人工确认       |
| 已完成     | `completed`      | 测试通过，代码已提交         |
| 失败       | `failed`         | 执行或测试失败，需人工介入   |
| 已取消     | `cancelled`      | 任务被手动取消               |

#### F-06 状态流转规则

```plaintext
pending → in_progress（触发执行）
in_progress → pending_review（Claude Code 返回成功）
in_progress → failed（Claude Code 返回失败）
pending_review → completed（测试通过，代码提交）
pending_review → failed（测试失败，生成 bug）
failed → pending（手动重置，重新执行）
pending / in_progress / pending_review → cancelled（手动取消）
```

---

### 3.4 任务执行

#### F-07 触发任务执行

用户在任务详情页点击「执行」，系统执行以下流程：

1. **创建 Git 分支**：命名规则 `feat/<task-id>-<简短标题>` 或 `fix/<task-id>-<简短标题>`，从 `dev` 分支切出
2. **创建 Git Worktree**：在项目目录下创建独立工作区 `.worktrees/<branch-name>/`，与主仓库隔离
3. **启动 Claude Code 窗口**：后端调用命令行在对应 worktree 路径下启动 Claude Code 进程，将任务描述（含验收标准）作为初始 Prompt 发送
4. **任务状态更新**：触发后立即将状态置为 `in_progress`

#### F-08 Prompt 构造规则

发送给 Claude Code 的内容包含：

- 任务类型和标题
- 任务完整描述
- 验收标准列表
- 当前分支和 worktree 路径说明
- 要求 Claude Code 在完成后明确返回执行结果（`SUCCESS` 或 `FAILURE: <原因>`）
- 若本项目存在 `docs/开发文档/经验总结.md`，将其内容附加，要求 Claude Code 参考规避已知问题

#### F-09 接收执行结果

- 后端监听 Claude Code 进程的标准输出，提取执行结果标记
- 执行成功（`SUCCESS`）：状态置为 `pending_review`
- 执行失败（`FAILURE`）：状态置为 `failed`，提取失败原因写入 `docs/开发文档/经验总结.md`（格式见 3.6），保留 worktree 以便排查

---

### 3.5 测试与提交

#### F-10 测试执行

任务进入 `pending_review` 状态后，系统提示用户执行测试：

- 用户可在界面上查看测试用例列表（从任务验收标准生成）
- 支持手动标记每条测试用例为「通过」或「失败」
- 后端可选支持自动运行项目配置的测试命令（如 `mvn test` / `npm test`）

#### F-11 测试通过 → 提交代码

所有测试用例通过后，系统自动执行：

1. 在 worktree 内提交代码：`git commit -m "feat: <任务标题>"`
2. 将分支推送到远端
3. 创建 PR/MR（可选，需配置 GitHub/GitLab token）
4. 清理 worktree：`git worktree remove <path>`
5. 任务状态置为 `completed`

#### F-12 测试失败 → 生成 Bug

有测试用例失败时：

1. 在项目 `docs/bug/` 目录下创建 Bug 报告文件
2. 文件命名规则：`<YYYY-MM-DD>-<模块>-<功能>-<简短描述>.md`
3. 使用以下模板生成 Bug 报告内容（见下方模板）
4. 自动创建一个新的 `bug` 类型任务，关联此 Bug 报告文件路径
5. 原任务状态置为 `failed`

Bug 报告内容模板：

```markdown
# Bug 报告

- **时间**：YYYY-MM-DD HH:mm
- **模块**：xxx
- **功能**：xxx
- **任务 ID**：uuid
- **分支**：feat/xxx

## 问题描述

（失败的测试用例描述）

## 复现步骤

（待填写）

## 期望结果

（验收标准原文）

## 实际结果

（测试失败的实际现象）

## 状态

- [ ] 待修复
```

---

### 3.6 经验总结归档

#### F-13 失败经验写入

当任务执行失败（Claude Code 返回 `FAILURE`）时，系统向 `docs/开发文档/经验总结.md` 追加以下格式内容：

```markdown
---

## <YYYY-MM-DD> <任务标题>

**问题**：<Claude Code 返回的失败原因>

**原因**：（待分析，可由后续人工补充）

**解决方案**：（待填写）
```

#### F-14 经验注入执行

每次触发任务执行时，若 `经验总结.md` 非空，将其内容作为上下文附加到 Prompt 中，要求 Claude Code 参考已有经验规避重复问题。

---

### 3.7 任务详情与日志

#### F-15 任务详情页

展示以下信息：

- 基本信息：ID、类型、标题、模块、状态、时间
- 任务描述（Markdown 渲染）
- 验收标准（可勾选）
- 关联分支和 worktree 路径
- 执行日志（Claude Code 输出的实时/历史日志）
- 关联 Bug 报告（若有）

#### F-16 执行日志

- 实时展示 Claude Code 的输出流
- 历史日志持久化存储，支持按任务查询

## 四、非功能需求

### 4.1 性能

- 任务清单解析：< 2 秒（文件 < 1MB）
- 任务状态变更响应：< 500ms
- Claude Code 执行为异步，不阻塞界面

### 4.2 可靠性

- Claude Code 进程异常退出时，任务状态自动标记为 `failed`
- worktree 创建失败时，回滚并提示错误原因

### 4.3 兼容性

- 操作系统：Windows 10+、macOS 12+、Linux（主流发行版）
- 依赖：本地已安装 Git（>= 2.17，支持 worktree）、Claude Code CLI

## 五、数据模型概览

| 实体           | 说明                         |
| -------------- | ---------------------------- |
| Project        | 项目，包含路径、名称、配置   |

| ExecutionLog   | 任务执行日志，关联 Task      |

### 5.2 任务类型枚举

| 值          | 说明         |
| ----------- | ------------ |
| `feature`   | 新功能开发   |
| `bug`       | 缺陷修复     |

### 5.3 任务状态枚举

| 值               | 说明       |
| ---------------- | ---------- |
| `pending`        | 待开发     |
| `in_progress`    | 开发中     |
| `pending_review` | 待 review  |
| `completed`      | 已完成     |
| `failed`         | 失败       |
| `cancelled`      | 已取消     |

## 六、界面需求（概要）

### 6.1 页面列表

| 页面       | 路由                               | 说明                           |
| ---------- | ---------------------------------- | ------------------------------ |
| 项目列表   | `/`                                | 所有项目的概览卡片             |
| 项目详情   | `/project/:id`                     | 项目下的任务看板（按状态分列） |
| 任务详情   | `/project/:id/task/:taskId`        | 任务完整信息与操作入口         |
| 执行日志   | `/project/:id/task/:taskId/log`    | Claude Code 实时/历史输出      |
| Bug 列表   | `/project/:id/bugs`                | 项目下所有 Bug 报告            |

### 6.2 任务看板

任务以看板形式按状态分列展示：

```plaintext
[ 待开发 ] [ 开发中 ] [ 待review ] [ 已完成 ] [ 失败 ] [ 已取消 ]
```

每个任务卡片展示：类型标签（feature/bug）、标题、模块、更新时间、快捷操作按钮。

## 七、范围边界

### 7.1 本期包含（v1.0）

- 项目添加与任务清单解析
- 任务状态流转与手动管理
- 触发 Claude Code 执行（本地进程）
- 执行结果接收与状态更新
- 失败经验写入 `经验总结.md`
- 测试用例手动标记
- 测试通过后代码提交
- 测试失败后 Bug 报告生成与 Bug 任务创建
- 任务执行日志查看

### 7.2 本期不包含

- 多人协作 / 权限管理
- 远端 PR/MR 自动创建（需额外配置）
- 自动化测试命令执行（手动标记优先）
- 任务清单自动生成（由 Claude Code 在项目中完成）
- 云端同步与数据备份

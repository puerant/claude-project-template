# 项目规范

## 语言
所有回复、注释、文档均使用**中文**。

## 项目结构

```
frontend/
  portal/     # 用户门户前端（Vue3 + TypeScript）
  admin/      # 管理后台前端（Vue3 + TypeScript）
backend/      # 后端服务（Java 17 + Spring Boot 3）
docs/
  需求文档/
    PRD.md              # 系统核心需求（只读参考）
  系统设计/              # 架构与模块设计（精简，不含实现代码）
  开发文档/
    任务清单.md          # 开发任务、排期与里程碑
    经验总结.md          # 踩坑记录与纠偏经验
```

## 工作流程

### 开始新功能前
1. 读 `docs/需求文档/PRD.md` 确认需求范围
2. 读 `docs/系统设计/` 了解已有设计
3. 读 `docs/开发文档/任务清单.md` 确认当前任务状态

### 设计阶段
- 在 `docs/系统设计/` 记录模块划分、接口契约、数据模型
- **不写实现代码**，保持设计文档精简

### 开发阶段
- 严格对齐 PRD 和系统设计，不自行扩展范围
- 每完成一个里程碑节点，更新 `docs/开发文档/任务清单.md`

### 发现问题时
- 偏离需求或走弯路后，在 `docs/开发文档/经验总结.md` 记录原因与纠正方案

## 编码规范

### 通用
- 函数职责单一，避免过度抽象
- 只处理当前需求，不为假设性需求预留扩展
- 不添加未被要求的错误处理、日志、注释

### 前端（Vue3 + TS）
- 使用 `<script setup>` 语法
- 状态管理优先 `composables`，复杂场景用 Pinia
- 接口类型定义放在 `src/types/`

### 后端（Java 17 + Spring Boot 3）
- 分层：Controller → Service → Repository
- 使用 Record 或 DTO 传递数据，不暴露实体
- 异常统一通过 `@RestControllerAdvice` 处理

## Git 规范

### 分支策略
- `main` — 生产就绪代码，只接受 PR 合并
- `dev` — 集成分支，功能完成后合并至此
- `feat/xxx` — 新功能，从 `dev` 切出
- `fix/xxx` — 缺陷修复，从 `dev` 切出

### 提交信息
格式：`<type>: <简短描述>`

| type | 用途 |
|------|------|
| feat | 新功能 |
| fix | 缺陷修复 |
| refactor | 重构 |
| docs | 文档变更 |
| chore | 构建/配置 |

示例：`feat: 用户登录接口` / `fix: 修复分页参数越界`

### 操作规则
- 不直接推送到 `main`，必须经过 PR/MR
- 不使用 `--force` 推送（紧急情况需确认）
- 不跳过 hooks（`--no-verify`）
- PR 合并前确保 `dev` 上已测试通过

## 文档维护原则

- `PRD.md`：**不修改**，仅作参考基线
- `系统设计/`：新增模块时补充，精简描述，无代码
- `任务清单.md`：任务状态变更时同步更新
- `经验总结.md`：每次纠偏后追加记录，格式：问题 → 原因 → 解决方案
